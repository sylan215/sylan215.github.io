简历中打动人的 3 个关键点

发表于 2018-08-29 | 分类于 测试相关 ， 职业发展 | | 热度 ℃

前几天推送《简历中千万不要出现的 3 个问题》后，不少朋友在后台告诉我，这几个提醒太关键太及时太有用了，但是，这么做只是避免简历被轻易涮掉，并没有提高简历通过率呀。

确实是这样，那怎样才能增加简历通过率呢？今天我就从筛简历的角度聊聊，简历中突出哪些软技能时，更有可能赢得面试机会。

我大概总结了一下，可以突出如下 3 点：

突出自己的主动性；
突出自己解决问题的能力；
突出自己的学习能力；

下面我就每一点分别展开说一下。

一、突出自己的主动性

《现代汉语词典》中解释，「主动」就是「不待外力推动而行动」，那么「主动性」就是这种主动行动的具体体现。

举个例子。

张三在他的「专业技能」一栏写了：

自学 Python 半年，目前为项目编写了 5 个测试工具；

我们先不管这 5 个工具到底有没有技术含量，首先看到「自学、半年、工具」，这几个字眼，是不是很吸引人？

现在已经是互联网发展的成熟期，借助互联网自学已经不是什么新鲜事，自学成才的人也都不在少数，但要是按比率算，能坚持半年，并且有产出的人，真的是很少，至少也是一个很可贵的品质了。

如果你正在准备自学，或者已经开始自学，甚至已经有产出了，请一定不要忘记在简历中体现出来。

我们再看一个例子。

张三在他的「自我总结」中提到：

曾抽空花 2 天时间，把新参与项目中的历史 Bug 全部看了一遍，对项目中可能出现问题的功能点，以及出现频度比较高的 Bug 类型都做了初步了解，从而很快就上手了新项目。

看到这个介绍，有没有怦然心动的感觉？刨去别的描述，如果简历中仅有这一条，我也是肯定会约的。

其他类似的主动性体现的东西，应该还有很多，工作中只要不仅仅是把事情做完，而是有意识的去把事情做好，甚至更好，在写简历时必然有很多亮点可以写的。

二、突出自己解决问题的能力

我说的解决问题的能力，可以分为两类，一类是自己动手解决问题，另一类是自己利用资源把问题解决了。

还是有点懵？

比如我之前写的《Python3.4 安装 pycrypto 提示错误的处理办法》和《恢复 Postman 中误删除的 Collection 的方法》，算是自己动手把问题解决了，如果简历中写了类似的经历，就体现了个人解决问题的能力。

至于另一点，什么叫利用资源解决问题？我们先看个问题：

如果一个平时需要一人 5 天才能测试完成的项目，现在因为需求紧急，项目方只给了 2 天的测试时间，你该怎么办？

面试时，我经常获得的答复包括：

优先跑优先级高的用例，其他用例根据时间酌情安排；
和项目方沟通，尽量多争取点时间；
自己加班加点，尽量赶进度，实在不行也没办法；

这几个答复有毛病么？没毛病，有考虑到用例优先级划分的，也有考虑到和项目方的沟通，但是看出来了么？三个方案的核心都是「我」在尽力处理「麻烦」。

我们来看看另一个人的答复：

看起来一个人肯定是搞不定了，需要找上级商量看看是否可以协调资源支持，搞个突击，同时也需要周知项目方关于时间缩短对项目质量造成的影响，看看他们是否可以接受，或者再多给点时间，如果都没问题，就按用例优先级划分，尽量往前赶进度。

如果和这个答复比起来，是不是就感觉前面考虑的太少了？这个增加了向上级求助，寻求资源支持的过程，也就是团队思维，同时也增加了周知项目方可能存在质量风险的过程，这是做事的思维，而不仅仅是解决手头的麻烦。

有人可能会说，有沟通的时间，我都多跑几条用例了，是，所以你才适合做用例执行的角色。

如果你在项目中，有上述的一些解决问题的表现，请尽量在简历中体现出来，不管别人约不约，我反正是约定了。

三、突出自己的学习能力

我这里说的学习能力，也是分两种类型的。

第一种类似在学校的学习，就是指定要学习的内容，根据花费的时间以及所达到的不同程度，来区分的学习能力。

比如：

张三之前是做 Web 测试的，因为项目需要调岗去支持
app 测试，经过三天的了解和学习，已经掌握了 App 测试的所有注意事项，并可以在项目中发挥中坚力量。

嗯，我们不细究张三究竟怎么做到的，仅从结果来说，这学习能力是杠杠的。

我说的另一种类型，强调的是通过复盘、总结、学习等方式达成经验积累的效果，简单说就是举一反三的能力。

比如：

张三在最新的 ARM64 系统上测试时，发现某个功能因为使用了函数 IsWow64Process 导致了功能异常，查资料发现 ARM64 系统上改为使用 IsWow64Process2 了。
这还不算完，他觉得整个项目中肯定还有其他地方有使用 IsWow64Process 这个 API，于是暴搜一把，果然如此，进而督促开发制定了统一的修改策略，既解决了当前出现的问题，又处理了其他可能存在同样隐患的地方，一举两得，还降低了整体的修复成本。
这还不算完，为了避免开发在新实现的代码中再次出现这个问题，他又督促代码审计系统增加了对这个 API 调用的检查，彻底杜绝该隐患。

就这么平铺直述起来，可能不少人觉得没什么大不了，可从我面试的情况看，有些人在发现问题后，根本不关心问题产生的原因，更别说举一反三、避免同类问题再次发生了。

我面试的还有不少人，自己写的代码，会被自己写的用例测的漏洞百出，而所有出现的 Bug 都是自己曾经报给开发同学的低级错误，是不是很尴尬？

所以，如果你在平时工作中有关于学习能力的具体体现，请务必在简历中体现出来，有很高的加分哦。

好了，我能帮你的就这么多了，全部都是经验之谈噢，赶紧去优化下你的简历投给我吧。
